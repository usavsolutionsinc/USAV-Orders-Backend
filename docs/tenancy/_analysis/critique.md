# Critic Punch List — Multi-Tenancy Hardening Plan (6 specs)

## VERDICT
The 6 specs are individually strong and repo-accurate. But **as a set they describe everything EXCEPT the one fix that makes any of it work**. Confirmed against the repo: `withTenantConnection`/`withTenantTransaction` correctly `set_config('app.current_org', …)` (`src/lib/tenancy/db.ts:53,93`), `enforce_tenant_isolation` does ENABLE+FORCE+policy but **never touches roles** (`2026-06-14_rls_enforcement_infra.sql` — grep for `BYPASSRLS`/`CREATE ROLE` returns nothing), and **no migration anywhere creates a non-bypassrls app role** (grep of `src/lib/migrations/` + `docs/` for `NOBYPASSRLS`/`CREATE ROLE`/`app_tenant` is empty). The app connects via `DATABASE_URL` with zero role-switch logic (`src/lib/db.ts:40`, `src/lib/drizzle/db.ts:9`). So every "REAL LEAK" fix in the routes/repos specs, and every "FORCE in Phase C" reference, is **load-bearing on a Phase E role flip that none of the 6 specs own.**

## (1) CONTRADICTIONS with the empirical finding (neondb_owner.BYPASSRLS defeats FORCE)

**C1 — `tables` spec, B6 & §1 framing: treats ENABLE+policy as the protection, never names BYPASSRLS.** B6 arms RLS on 23 tables and the §1 rationale says batches "keep working" for raw-pool routes because RLS is non-forced. This is technically true but **silently reinforces the false mental model that ENABLE/FORCE is the security boundary.** It is not — BYPASSRLS makes both inert. Correction: every B-batch header comment that says "ENABLE RLS + policy" must add: *"INERT until the app role loses BYPASSRLS (Phase E). This migration provides correctness scaffolding only; it grants ZERO isolation on its own."* Otherwise an operator ships B1–B6, sees policies in `pg_policies`, and believes tenants are isolated when they are 100% exposed.

**C2 — `cron` spec §5.1 is the ONLY spec that gets this right, and it half-states it.** It correctly says "until the cron runtime DSN uses a non-BYPASSRLS role, the GUC does nothing" and proposes a two-pool split (`adminPool` owner + tenant `pool`). **But it frames the role flip as a cron precondition rather than a global one.** Correction: promote §5.1's role observation to a **Phase E that ALL six specs depend on**, and note the two-pool split is needed app-wide (not just cron) because `enforce_tenant_isolation` also FORCEs `roles`/`organizations` — the role-store loader and the `forEachActiveOrg` enumeration both need the privileged pool.

**C3 — `routes` spec "Shared fact #4" is correct but the fixes don't sequence against it.** It correctly states "RLS is inert (BYPASSRLS on neondb_owner), so the only thing protecting these tables today is an explicit WHERE." Good. **The contradiction is implicit:** the fixes lean on `withTenantTransaction` so the "usav-fallback INSERT default resolves to the real tenant" (§2, §3, §6, §10). That is true and works under BYPASSRLS (it's just `set_config`, not RLS). But the **SELECT-side** fixes depend on the explicit `WHERE organization_id = $n` — which means **these routes must NOT rely on RLS at all until Phase E**, and after Phase E the explicit WHERE becomes belt-and-suspenders. The spec should state that the explicit WHERE clauses are the *permanent* primary guard for any neon-http path (which can never carry the GUC), per the `repos` spec's Option C reality.

**C4 — No spec states the FORCE/BYPASSRLS interaction for the `repos` Option A1 conclusion.** The `repos` spec correctly says neon-http can't carry the GUC and recommends `withTenantDrizzle` over the WS pool. **But its step 5 ("run `enforce_tenant_isolation()` … after the app's non-bypassrls connection role is in place") is the only mention of the role flip in that spec, and it's buried.** It must be elevated: even after migrating to WS + GUC, FORCE does nothing while the role is BYPASSRLS. The WS migration is necessary but NOT sufficient.

**Net on (1):** No spec proposes "keep neondb_owner + FORCE" — so there is no direct contradiction with the empirical finding. The defect is **omission**: only the cron spec names the role flip, and even it scopes it too narrowly. **Phase E (create a `NOBYPASSRLS`, non-owner app role; flip the runtime `DATABASE_URL`; keep a separate privileged pool) is the missing keystone and must be authored as its own spec that all others list as a hard dependency.**

## (3) SINGLE HIGHEST-RISK THING TO SHIP FIRST

**Ship the Phase E role infrastructure FIRST — but as a paired commit with the role-store fix — gated by the cross-org canary test.** Concretely, in order:

1. **`role-store.ts` org-keying (`infra` D4d).** This is the one **active, exploitable cross-tenant leak that exists TODAY, with zero new infra required** — under warm Lambdas, org B's request gets org A's `roles`/permissions because `fetchRoles` (`role-store.ts:50`) has no org filter and the cache is process-wide. It's an RBAC/privilege-escalation leak, not a future risk. It needs no role flip to fix and no role flip to be dangerous. **This is the only finding that is a live vulnerability rather than a latent one.**

2. **Phase E role + two-pool split + the BYPASSRLS canary test**, because every other "fix" in the routes/repos/cron specs is **inert theater until the runtime role is non-bypassrls.** Shipping the 13 route fixes or the cron sweep first gives a false sense of security: the explicit `WHERE organization_id` clauses help, but any un-migrated raw-pool path (300+ of them) still leaks, and RLS provides no backstop. The role flip + canary (`a raw pool.query with no GUC returns zero rows on a FORCEd table`) is what converts RLS from inert to load-bearing and makes the explicit-WHERE work a defense-in-depth layer rather than the sole guard.

Rationale for the pairing: do #1 immediately (it's a real bug, independently shippable). Do #2 as the keystone the rest of the plan hangs on. Everything else (tables B1–B6, routes, repos, realtime, cron) is correctly specified but must be sequenced *behind* a written Phase E spec — which is the plan's largest missing piece.
