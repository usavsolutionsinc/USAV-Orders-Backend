# Node-Workflow Engine — Detailed Implementation Plan

> **Companion to:** `docs/operations-studio/NODE_WORKFLOW_ARCHITECTURE.md` (the why/what). This doc is the **how** — file-by-file, in dependency order, MVP-first.
> **Goal:** Build the visual node-graph "Operations" layer and surface it by **replacing the admin Architecture page** with a live workflow canvas.
> **Constraint:** Reuse existing domain modules; add only the graph layer, engine, and canvas. Work on `main` (per project convention).

---

## 0. The "replace the Architecture page" decision

The current **Architecture** admin section is a *developer* tool — it renders static codebase Mermaid diagrams (`module deps`, `routes`, `recent commits`) generated by a post-commit hook and served from `/api/architecture` reading `docs/architecture/*.mmd`. It is **not** business data, and it still works.

**Recommended approach — repurpose, don't destroy:**

| Slot | Before | After |
|---|---|---|
| `architecture` section | Codebase diagram viewer | **Operations canvas** (React Flow node graph) |
| *new* `codebase` section | — | The old codebase viewer, relabeled & moved to the bottom of `System` (dev-only via permission) |

This satisfies "replace the Architecture page" literally — that slot becomes the operations graph — while preserving a working tool (zero data loss, the `/api/architecture` endpoint, generator, and post-commit hook are untouched; only the section key/label/components it mounts under change).

> If you'd rather **fully remove** the codebase viewer instead of relocating it, see §9 "Alternative: hard removal." Default plan below assumes relocate.

## 1. Dependency order (build bottom-up)

```
Phase A  DB schema (graph tables)                 ── no UI, no risk
Phase B  Engine core (contract/registry/runtime)  ── pure TS, unit-testable
Phase C  Node adapters over existing modules      ── thin wrappers
Phase D  API routes (/api/workflow/*)             ── guarded, validated
Phase E  React Flow canvas + admin slot swap       ── the visible replacement
Phase F  Live item tracking (Ably) + routing       ── the "wow"
Phase G  Editable canvas + templates + perms       ── operator self-serve
```

Each phase is independently shippable and leaves the app green. Phases A–D touch **no** existing operational code. Phase E is the first user-visible change (the admin page swap).

## 2. Phase A — Database schema

**File:** `src/lib/drizzle/schema.ts` (append). Then `npm run db:generate && npm run db:migrate`.

```ts
// ─── Workflow graph layer ────────────────────────────────────
export const workflowDefinitions = pgTable('workflow_definitions', {
  id: serial('id').primaryKey(),
  organizationId: uuid('organization_id').notNull(),
  name: text('name').notNull(),
  version: integer('version').notNull().default(1),
  isActive: boolean('is_active').notNull().default(false),
  createdBy: integer('created_by').references(() => staff.id),
  createdAt: timestamp('created_at').defaultNow(),
  updatedAt: timestamp('updated_at').defaultNow(),
});

export const workflowNodes = pgTable('workflow_nodes', {
  id: text('id').primaryKey(),                       // canvas node uuid
  workflowDefinitionId: integer('workflow_definition_id')
    .notNull().references(() => workflowDefinitions.id, { onDelete: 'cascade' }),
  type: text('type').notNull(),                      // registry key
  positionX: real('position_x').notNull(),
  positionY: real('position_y').notNull(),
  config: jsonb('config').notNull().default({}),
});

export const workflowEdges = pgTable('workflow_edges', {
  id: text('id').primaryKey(),
  workflowDefinitionId: integer('workflow_definition_id')
    .notNull().references(() => workflowDefinitions.id, { onDelete: 'cascade' }),
  sourceNode: text('source_node').notNull(),
  sourcePort: text('source_port').notNull(),         // output port id → conditional routing
  targetNode: text('target_node').notNull(),
});

export const itemWorkflowState = pgTable('item_workflow_state', {
  id: serial('id').primaryKey(),
  organizationId: uuid('organization_id').notNull(),
  serialUnitId: integer('serial_unit_id').notNull().references(() => serialUnits.id),
  workflowDefinitionId: integer('workflow_definition_id')
    .notNull().references(() => workflowDefinitions.id),
  currentNodeId: text('current_node_id').notNull(),
  status: text('status').notNull().default('active'), // active | blocked | done | error
  context: jsonb('context').notNull().default({}),    // accumulated node outputs
  enteredNodeAt: timestamp('entered_node_at').defaultNow(),
  updatedAt: timestamp('updated_at').defaultNow(),
}, (t) => ({ oneActivePerUnit: uniqueIndex('iws_active_unit').on(t.serialUnitId) }));

export const workflowRuns = pgTable('workflow_runs', {  // observability, mirrors pipeline_cycles
  id: serial('id').primaryKey(),
  organizationId: uuid('organization_id').notNull(),
  serialUnitId: integer('serial_unit_id').notNull(),
  workflowDefinitionId: integer('workflow_definition_id'),
  nodeType: text('node_type').notNull(),
  output: text('output'),
  durationMs: integer('duration_ms'),
  error: text('error'),
  createdAt: timestamp('created_at').defaultNow(),
});
```

**Checklist**
- [ ] Append tables; confirm `uuid`, `real`, `jsonb`, `uniqueIndex` are imported in `schema.ts`.
- [ ] `npm run db:generate` → review the generated SQL migration.
- [ ] `npm run db:migrate:dry` then `npm run db:migrate`.
- [ ] `npm run db:studio` — eyeball the 5 new tables.

**Acceptance:** migration applies cleanly; no existing table altered.

## 3. Phase B — Engine core

**New dir:** `src/lib/workflow/`. Pure TypeScript; no Next.js, no React. Unit-test with `tsx --test` (the repo's existing pattern, e.g. `src/lib/auth/*.test.ts`).

| File | Responsibility |
|---|---|
| `contract.ts` | `NodeDefinition`, `NodeContext`, `NodeResult`, `WorkflowEvent` (see architecture doc §3.2) |
| `registry.ts` | `registerNode` / `getNode` / `listNodes` |
| `router.ts` | `resolveNext(defId, nodeId, port)` → reads `workflow_edges` |
| `runtime.ts` | `runNode(def, ctx)` — times it, writes `workflow_runs`, catches errors |
| `advance.ts` | `advanceItem(...)` — load state → run → route → move pointer (architecture doc §3.5) |
| `events.ts` | `emit()` → Ably publish + `inventory_events` insert (reuse `src/lib/realtime`) |
| `index.ts` | `registerBuiltins()` + public exports |

**Checklist**
- [ ] `contract.ts` — copy types from architecture doc §3.2; wire `OrgId` from `@/lib/tenancy/constants`.
- [ ] `registry.ts` — copy from architecture doc §3.3.
- [ ] `router.ts` — single Drizzle query on `workflowEdges` by `(workflowDefinitionId, sourceNode, sourcePort)`.
- [ ] `runtime.ts` — wrap `def.run(ctx)` in try/catch + `Date.now()` timing → insert `workflowRuns`.
- [ ] `advance.ts` — wrap body in `withTenantConnection(orgId, …)`; acquire Upstash lock on `serialUnitId` (reuse existing Upstash client) to prevent double-advance.
- [ ] `events.ts` — reuse `src/lib/realtime` Ably publisher + insert `inventoryEvents`.
- [ ] **Tests:** `src/lib/workflow/router.test.ts`, `advance.test.ts` with an in-memory fake registry + stubbed db.

**Acceptance:** `tsx --test src/lib/workflow/*.test.ts` green; engine advances a fake item across a 3-node linear graph and a fail→repair branch.

## 4. Phase C — Node adapters (thin wrappers over existing modules)

**New dir:** `src/lib/workflow/nodes/`. Each file ≈30 lines and calls an existing `src/lib/*` function. **Do not reimplement domain logic.** Before writing each adapter, find the existing entry point:

| Node | Wraps (verify exact export) |
|---|---|
| `receiving.node.ts` | `src/lib/receiving` (intake/receive line) |
| `inspection.node.ts` | `src/lib/tech` (record test result → `testing_results`) |
| `repair.node.ts` | `src/lib/repair` (`repair_service` / issue templates) |
| `listing.node.ts` | `src/lib/ebay` (create/update offer) |
| `pricing.node.ts` | `src/lib/inventory` / `sku_catalog` |
| `order.node.ts` | `src/lib/orders` / `orders-sync` |
| `pack-ship.node.ts` | `src/lib/picking` + `src/lib/shipping` |
| `returns.node.ts` | `src/lib/rma` / `orders_exceptions` |
| `conditional.node.ts` | pure logic (no side effects) — evaluates `config.rule` against `ctx.context` |

**Process per node:** (1) `grep` the module's `index.ts` for the public function, (2) write the adapter mapping `NodeContext → that function's args`, (3) map its result to a `NodeResult.output` port, (4) register it.

**Checklist**
- [ ] Implement `receiving`, `inspection`, `pack-ship`, `conditional` first (covers the MVP linear flow + one branch).
- [ ] `registerBuiltins()` imports all node files so their `registerNode()` side-effects run.
- [ ] Add `repair`, `returns`, `listing` in Phase F/G as routing matures.

**Acceptance:** `listNodes()` returns the registered set with `label`/`icon`/`outputs` for the palette.

## 5. Phase D — API routes

**New dir:** `src/app/api/workflow/`. Follow the repo's route conventions (the `api-route-reviewer` agent checks: **auth guard + Zod validation + idempotency on mutations + audit-log emit**).

| Route | Method | Purpose |
|---|---|---|
| `nodes/route.ts` | GET | `listNodes()` → palette metadata |
| `definitions/route.ts` | GET/POST | list / create workflow definitions |
| `definitions/[id]/route.ts` | GET/PUT | load / save graph (nodes+edges) |
| `definitions/[id]/activate/route.ts` | POST | publish a version (`isActive`) |
| `advance/route.ts` | POST | `{ serialUnitId, trigger }` → `advanceItem()` (idempotency-keyed) |
| `state/[serialUnitId]/route.ts` | GET | current node + run history for an item |

**Checklist**
- [ ] Every handler: `await requirePermission('workflow.view' | 'workflow.manage', …)` (Phase D.1 below adds these).
- [ ] Zod-validate bodies (reuse `src/lib/schemas` patterns).
- [ ] `advance` mutation: idempotency key = `${serialUnitId}:${trigger.hash}`; audit-log emit on transition.
- [ ] PUT `definitions/[id]`: replace nodes/edges in a transaction.
- [ ] Run the `api-route-reviewer` agent over the new routes.

## 7. Phase F — Live item tracking + conditional routing

- **Tracking:** `ItemTracker` overlay subscribes to Ably (reuse `src/lib/realtime`); on a `WorkflowEvent`, animate a dot to the node a scanned serial unit entered. Backfill `item_workflow_state` from the latest `station_activity_logs` row per unit (one migration script in `scripts/`).
- **Routing live:** wire the scan trigger → `/api/workflow/advance`; enable `inspection.fail → repair`, `repair.done → inspection` rework loop.
- **Add nodes:** `repair`, `returns`.

**Acceptance:** scanning a unit on the floor moves its dot on the board in real time; a failed inspection auto-parks the unit at Repair.

## 8. Phase G — Editable canvas, templates, team perms

- **Edit/save:** drag from palette, connect ports, per-node config form from `configSchema`; `PUT /api/workflow/definitions/[id]` (gated `workflow.manage`). Versioning: publishing a new version leaves in-flight items on their old version.
- **Templates:** seed 2–3 (`Standard refurb-and-list`, `Test-only consignment`, `Returns triage`) as `workflow_definitions` rows; empty-state onboarding via the `onboard`/design-system patterns.
- **Team:** per-node permission checks via `permission-registry`; activate `organization_id`/RLS path that `src/lib/tenancy` already stubs; assign nodes to `staff_stations` / `work_assignments`.

**Acceptance:** operator rewires the flow and publishes without a deploy; new users start from a template.
